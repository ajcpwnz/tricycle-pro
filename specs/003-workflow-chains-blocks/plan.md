# Implementation Plan: Workflow Chains & Pluggable Blocks

**Branch**: `003-workflow-chains-blocks` | **Date**: 2026-03-24 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/003-workflow-chains-blocks/spec.md`

## Summary

Replace the hardcoded specify → plan → tasks → implement workflow with a configurable chain system supporting three variants, and decompose all command behavior into pluggable blocks (partial system prompts) that users can toggle via `tricycle.config.yml`. A new `tricycle assemble` command generates command files from enabled blocks. Build-time assembly means Claude Code reads static files — zero runtime overhead.

## Technical Context

**Language/Version**: Bash 3.2+ (macOS default), 4.0+ (Linux)
**Primary Dependencies**: Standard Unix utilities (`sed`, `awk`, `grep`, `find`, `cat`, `mkdir`)

**Testing**: `node --test` (existing test infrastructure in `tests/`)
**Target Platform**: macOS / Linux (CLI tool)
**Project Type**: CLI toolkit
**Performance Goals**: Assembly completes in <2 seconds for any chain/block configuration
**Constraints**: No Node.js runtime for assembly (pure bash); Bash 3.2 compatibility for macOS
**Scale/Scope**: ~22 built-in block files, 5 assembled commands, 1 new CLI subcommand

## Key Implementation Notes

### Assembly Flow

1. `tricycle assemble` reads `tricycle.config.yml` `workflow` section
2. For each chain step: collect blocks → apply overrides → add absorbed blocks → sort by order → concatenate → write command file
3. For omitted steps: write "blocked" stub command file
4. For headless: generate from chain (dynamic phase count)
5. Update `.tricycle.lock` checksums

### Absorption Logic

When a step is omitted, its non-required default blocks get absorbed into the **preceding step in the canonical chain order**:
- Omit tasks → tasks blocks absorbed into plan
- Omit plan+tasks → plan blocks absorbed into specify, then tasks blocks absorbed into specify
- Absorbed blocks get order values offset by +100 per absorption level to sort after the absorbing step's own blocks

### Command File Generation

Each assembled command file contains:
1. YAML frontmatter: `description` (generated), `handoffs` (next step in chain)
2. `## User Input` section with `$ARGUMENTS` reference
3. Concatenated block content (in order)

### Chain Validation Block

Every chain step gets a `chain-validation` block (order: 20, required) that:
1. Reads `tricycle.config.yml`
2. Checks if the current step is in `workflow.chain`
3. If not present: outputs error message and stops

### Input Validation Block

The specify step's `input-validation` block (order: 30, required) contains:
- Full chain: no minimum detail required
- 3-step chain: prompt must describe scope + outcomes
- 2-step chain: prompt must describe scope, behavior, constraints, acceptance criteria

### Headless Assembly

Headless command is assembled per-chain:
- Phase numbering matches chain length (Phase 1/2, 1/3, 1/4)
- Each phase invokes the corresponding chain step command
- Headless overrides (auto-continue, auto-resolve) stay inline
- Pause rules unchanged

### Extensions.yml Deprecation

- Remove all `extensions.yml` checking from tasks and implement command blocks
- No migration script — users manually convert extension hooks to custom blocks
- Document migration path in quickstart.md

### Files NOT Modified

- `trc.clarify.md`, `trc.analyze.md`, `trc.checklist.md`, `trc.constitution.md`, `trc.taskstoissues.md` — not chain steps, remain independent
- Enforcement hooks (`block-spec-in-main.sh`, `block-branch-in-main.sh`, `post-implement-lint.sh`) — untouched
- Templates (`spec-template.md`, `plan-template.md`, etc.) — unchanged
- Existing scripts (`create-new-feature.sh`, `check-prerequisites.sh`, etc.) — unchanged
